# Development

Most of this repo is generated by [Speakeasy](https://www.speakeasy.com/) from
`automation-api-oas.yaml` and the overlays in `.speakeasy/overlays/`. Day-
to-day work flows through `make speakeasy` (regenerate) and `make lint`
(validate the merged OAS); generated files are not edited by hand.

## Prerequisites

| Tool | Version | Notes |
|---|---|---|
| Go | matches `go.mod` (≥1.25) | `brew install go` |
| [Speakeasy CLI](https://www.speakeasy.com/docs/speakeasy-reference/cli/getting-started) | matches `.speakeasy/workflow.yaml`'s `speakeasyVersion` | `brew install speakeasy-api/homebrew-tap/speakeasy` |
| Terraform CLI | any recent (used by `terraform fmt`) | `brew tap hashicorp/tap && brew install hashicorp/tap/terraform` |
| Speakeasy account / API key | one-time | `speakeasy auth login` populates `~/.speakeasy/config.yaml` |

## Quick start

```sh
# one-time
speakeasy auth login

# regenerate the provider, SDK, docs, examples
make speakeasy

# lint the OAS-plus-overlays and check Terraform fmt
make lint

# unit tests (no AM running needed)
make unit-tests
```

`make help` lists every target.

## Editing generated code

Don't. Anything carrying a `// Code generated by Speakeasy` header is
overwritten on the next `make speakeasy`. To change provider behaviour, add
or update an overlay under `.speakeasy/overlays/` and regenerate. Useful
overlay annotations are documented in `.claude/CLAUDE.md`.

The merged OAS (`.speakeasy/output/computed.json`, gitignored) is rebuilt by
`make merge-overlays` and is the most accurate view of what Speakeasy
actually generates from. Open it when an overlay isn't behaving.

## Debugging the provider locally

Build once and let Terraform attach to your binary instead of fetching from
the registry. Two flavors:

### `--debug` reattach (good for delve / IDE debuggers)

```sh
go run main.go --debug
# Copy the exported TF_REATTACH_PROVIDERS line
# In another terminal:
cd examples/resources/am_domain
TF_REATTACH_PROVIDERS=... terraform init
TF_REATTACH_PROVIDERS=... terraform plan
```

### `dev_overrides` in `~/.terraformrc` (good for quick CLI runs)

```sh
go build  # produces ./terraform-provider-am
```

```hcl
# ~/.terraformrc
provider_installation {
  dev_overrides {
    "registry.terraform.io/gravitee-io/am" = "/absolute/path/to/terraform-provider-am-repo"
  }
  direct {}
}
```

With `dev_overrides` in place, **skip `terraform init`** for any examples
you run — Terraform will route to your local binary directly.

## Running against a live AM

The provider talks to AM's **Automation API**. To exercise it end-to-end you
need a running AM Management API with automation enabled, a bearer token, and
the provider pointed at it.

### Prerequisites

- A running AM Management API with the Automation API enabled. In
  `gravitee.yml`:
  ```yaml
  http:
    api:
      automation:
        enabled: true
  ```
- The provider built locally (`go build`) and wired via `dev_overrides` (see
  above).
- `jq` and `curl` for minting a token.

### Mint a bearer token

The default install ships an `admin` user (password `adminadmin`) and an
organization + environment both with id `DEFAULT`:

```sh
export AM_SA_TOKEN=$(curl -s http://localhost:8093/management/auth/token \
  -u admin:adminadmin \
  -d "grant_type=password&username=admin&password=adminadmin" | jq -r '.access_token')
```

For non-dev use, prefer a service-account access token over the admin password.

### Point the provider at AM

```sh
export AM_SERVER_URL=http://localhost:8093/automation
# AM_ORG_ID / AM_ENV_ID default to DEFAULT — override if required
```

### Apply an example

Runnable use-cases live under `examples/use-cases/` — each pairs a config
with a `provider.tf` declaring `source = "gravitee-io/am"`:

```sh
cd examples/use-cases/domain-simple
terraform plan      # no `terraform init` needed with dev_overrides
terraform apply
```

Confirm the resource landed via the raw API:

```sh
curl -s "$AM_SERVER_URL/organizations/DEFAULT/environments/DEFAULT/domains" \
  -H "Authorization: Bearer $AM_SA_TOKEN" | jq '.[].key'
```

For raw Automation API context (request/response shapes, the `key` model, the
`default` flag), see the AM repo's [WALKTHROUGH.md](https://github.com/gravitee-io/gravitee-access-management/blob/master/gravitee-am-management-api/gravitee-am-management-api-automation/WALKTHROUGH.md).

## Releasing

Releases run from `.github/workflows/release.yaml`, triggered manually with
a `dry_run` toggle (default `true`). The workflow reads the next version
from `.speakeasy/gen.lock` (`management.releaseVersion`), tags it, and runs
GoReleaser. Real releases need the `terraform_gpg_secret_key` and
`terraform_gpg_passphrase` repo secrets configured.
